# 폰트 생성 서비스 프로젝트 기획서

> 사용자가 직접 그린 글씨/그림에서 스타일을 추출해 전체 폰트를 자동 생성하는 웹 서비스. 웹앱 우선 출시 → 검증 후 토스 미니앱 전환을 목표로 한다.

---

## 1. 프로젝트 개요

### 1.1 서비스 핵심

사용자가 글자 몇 개를 입력(그리기 또는 업로드)하면, 그 스타일을 분석해 나머지 모든 문자를 동일한 스타일로 자동 생성해주는 폰트 제작 서비스다. 사용자는 굵기, 각도, 곡률 같은 파라미터를 슬라이더로 조절하며 실시간으로 결과를 미리 보고, 마음에 드는 시점에 전체 폰트 파일로 내보낸다.

### 1.2 핵심 가치

- **손이 덜 가는 폰트 제작**: 전문가가 수개월 그리던 작업을 몇 글자 입력으로 단축
- **실시간 파라미터 조절**: 슬라이더를 움직이면 글자 모양이 즉시 변하는 인터랙티브 경험
- **두 가지 생성 방식 병행**: 비용 없는 전통(알고리즘) 방식 + 고품질 AI 방식

### 1.3 단계별 전략

| 단계 | 내용 |
|------|------|
| Phase 1 | 메인 홈페이지 + 폰트 웹앱 동시(병렬) 개발, 두 생성 방식 함께 구현 |
| Phase 2 | 통합·테스트, 같은 입력으로 두 방식 결과 비교, 사용자 반응 수집 |
| Phase 3 | 트래픽/성과 보고 독립 도메인 분리 검토, 이모티콘·문자 생성 등 기능 확장 |
| Phase 4 | 충분히 검증되면 토스 미니앱 전환 (결제/정책 대응) |

## 2. 기술 스택

### 2.1 선택 기준

혼자 운영 + Oracle 무료 티어 + 비용 최소화. 이 세 가지를 모두 만족하는 가벼운 구성을 우선한다. 기존 사이트(tripavi.kr)가 Node.js/Python 커스텀 구조이므로 그 경험을 그대로 활용한다.

### 2.2 권장 스택

| 영역 | 기술 | 이유 |
|------|------|------|
| 프론트엔드 | Next.js (React 기반) | SEO에 강한 SSR/SSG 지원, 페이지별 메타태그 관리 용이 |
| 캔버스/드로잉 | HTML5 Canvas + Fabric.js | 사용자 글씨 입력 + 실시간 프리뷰 |
| 백엔드 | Node.js (Express 또는 Next API Routes) | 기존 경험 활용, 가벼움 |
| 폰트 생성 엔진 | Python (FastAPI) | 폰트/이미지 라이브러리 생태계가 Python에 집중됨 |
| 폰트 처리 | FontTools, Potrace, OpenCV | 윤곽선 추출·벡터화·폰트 파일 생성 |
| DB | PostgreSQL 또는 SQLite | 초기엔 SQLite로 충분, 확장 시 Postgres |
| 서버 | Oracle Cloud 무료 티어 | 비용 제로로 시작 |

> Node와 Python을 분리한 이유: 웹/API는 Node가 빠르고, 실제 폰트 변환 로직은 Python 라이브러리가 압도적으로 풍부하기 때문. 둘은 내부 HTTP로 통신한다.

## 3. 시스템 아키텍처

```
[사용자 브라우저]
      │  (글씨 입력, 파라미터 조절)
      ▼
[Next.js 프론트엔드]  ── SEO 메타태그, 실시간 프리뷰(Canvas)
      │  API 요청
      ▼
[Node.js 백엔드 / API Gateway]  ── 인증, 요청 라우팅, 결제(추후)
      │
      ├──▶ [Python 폰트 엔진 - 전통 방식]  비용 0, 로컬 처리
      │
      └──▶ [Python 폰트 엔진 - AI 방식]   개발: 로컬/CLI, 운영: 외부 API
                                          (유료 사용자에게만 노출)
      │
      ▼
[DB] 사용자 폰트 데이터·설정 저장
[스토리지] 생성된 폰트 파일(WOFF/TTF) 보관
```

## 4. 폰트 생성 엔진 (핵심)

### 4.1 전체 흐름 (5단계)

1. **입력 인식**: 사용자가 그린 글씨/그림을 이미지로 받음
2. **글자 분리**: 각 글자·획을 개별적으로 분리(세그멘테이션)
3. **벡터화(글리프 변환)**: 비트맵을 폰트용 윤곽선(벡터)으로 변환
4. **스타일 적용·보간**: 추출한 특징(굵기/각도/곡률)을 나머지 문자에 적용
5. **폰트 파일 인코딩**: WOFF(웹용) → 필요 시 TTF/OTF로 변환

### 4.2 두 가지 생성 방식

**(A) 전통(알고리즘) 방식 — 기본·무료**
- 입력 글씨에서 획 특징을 수학적으로 추출 → 규칙 기반으로 나머지 문자 생성
- 방식: 벡터 기반 특징 추출 + 보간(interpolation)
- 장점: 비용 제로, 빠름, 결과 예측 가능, 서버 부담 적음
- 단점: 복잡하거나 개성 강한 필체는 어색할 수 있음
- 적합: MVP, 무료 사용자

**(B) AI 방식 — 프리미엄·유료**
- 신경망이 스타일을 학습해 더 자연스럽게 생성(Style Transfer 계열)
- 개발/테스트: Claude CLI 또는 로컬 오픈소스 모델로 비용 없이 검증
- 실제 운영: Claude API 또는 Google API 연결, **사용한 사용자에게만 비용 청구**
- 장점: 품질 높음, 자연스러움
- 단점: API 비용 발생, (로컬 모델일 경우) 속도 느림·GPU 필요

> **AI 표시 의무 주의**: AI 방식으로 생성한 결과물에는 "AI로 생성됨" 라벨/배지/워터마크를 반드시 표시해야 한다. (특히 토스 미니앱 전환 시 필수 — 위반 시 과태료 리스크)

### 4.3 오픈소스 모델 관련 현실 점검

- 로컬 오픈소스 모델은 **무료지만 GPU 없으면 처리에 10초~1분**이 걸릴 수 있고, 품질도 상용 API보다 낮은 편
- 따라서 권장 전략: **개발·테스트는 오픈소스/CLI로, 실제 고품질 제공은 유료 API로**
- 무료(전통) → 유료(AI) 전환 시 품질·속도 차이가 자연스러운 업셀 포인트가 됨

## 5. 출력 포맷

| 포맷 | 용도 | 특징 |
|------|------|------|
| WOFF | 웹·앱 표시용 (초기 MVP 권장) | 파일 작고 배포 간단, 웹에서 바로 미리보기 |
| TTF/OTF | 사용자 PC 설치·범용 사용 | 어디서나 사용 가능, 다운로드 상품화 가능 |

> 초기엔 WOFF로 빠르게 구현하고, TTF/OTF 변환은 다운로드(유료) 기능으로 후속 제공.

## 6. SEO / 검색 노출 전략

### 6.1 핵심 결정: 단계적 도메인 전략

검색 노출을 위해 처음부터 독립 도메인을 여러 개 쪼개면 중복 콘텐츠 문제로 오히려 순위가 떨어질 수 있다. 그래서 다음 순서를 따른다.

1. **1단계 (검증)**: 메인 홈페이지 안에 서브페이지로 운영 — 예: `메인도메인/font`
   - 메인 도메인의 검색 신뢰도를 그대로 물려받음
   - 내부 링크 구조로 크롤러 순환이 좋아짐
2. **2단계 (확장)**: 폰트 앱이 충분히 성장하면 독립 서브도메인으로 분리 — 예: `font.메인도메인`
   - 마이그레이션 시 301 리다이렉트 + canonical 태그로 중복 방지

> 동시에 같은 콘텐츠를 `/font`와 `font.도메인` 양쪽에 올리면 검색엔진이 원본을 혼동한다. 반드시 한쪽만 노출하고 canonical로 정리.

### 6.2 페이지별 SEO

- 각 웹앱 페이지마다 고유 메타태그 설정 ("글씨체 만들기", "자동 폰트 생성", "손글씨 폰트" 등 키워드)
- Next.js SSR/SSG로 검색엔진이 콘텐츠를 잘 읽도록 함
- 구조화 데이터(JSON-LD), sitemap.xml, robots.txt 기본 세팅

## 7. 수익화 모델 (추후 적용)

> 기술 검증이 우선이므로 수익화는 Phase 2 이후 적용. 구조만 미리 설계해 둔다.

| 모델 | 설명 | 장단점 |
|------|------|--------|
| 사용량 기반 | 폰트 1개 생성/다운로드마다 결제 | 투명함 / 매번 결제 마찰 |
| 구독제 | 월정액 무제한 생성 | 사용자 편함 / 생성량 예측 어려움 |
| 프리미엄 기능 | 기본 무료 + 상업 라이선스·고급편집 유료 | 진입장벽 낮음 |
| 광고 | 무료 제공 + 광고 노출 | 진입장벽 가장 낮음 / 수익 단가 낮음 |

**권장 조합**: 전통 방식은 무료(+광고), AI 방식은 유료(사용량 또는 구독). AI API 비용을 유료 사용자에게 전가하는 구조.

## 9. 개발 단계별 마일스톤 (날짜 미지정)

### Phase 1 — 기초 구축 (병렬 진행)
- [ ] Oracle 서버에 새 환경/서브경로 세팅
- [ ] Next.js 메인 홈페이지 기본 틀 (레이아웃, 네비, SEO 기본)
- [ ] 폰트 웹앱: 드로잉 캔버스 + 파라미터 패널 + 실시간 프리뷰
- [ ] Python 엔진: 입력 인식 → 글자 분리 → 벡터화 → 보간 → WOFF 출력 (전통 방식)
- [ ] AI 방식: 로컬/CLI로 프로토타입 (비용 0)

### Phase 2 — 통합·테스트
- [ ] 폰트 앱을 `메인도메인/font` 서브페이지로 연동
- [ ] 같은 입력으로 전통 vs AI 결과 비교 화면
- [ ] 사용자 테스트, 피드백 수집, 품질·속도 측정
- [ ] AI 결과물 "AI 생성" 표시 적용

### Phase 3 — 최적화·확장
- [ ] 성과 분석 후 독립 도메인 분리 검토(301 + canonical)
- [ ] TTF/OTF 다운로드 기능
- [ ] 이모티콘·문자 이모티콘 자동 생성 기능 추가
- [ ] 수익화 모델 적용 (유료 AI / 광고)

### Phase 4 — 토스 미니앱 전환
- [ ] 토스 정책 점검 (AI 표시 의무, 외부 링크 제한, 인앱결제)
- [ ] 미니앱 내 완결형 구조로 재구성 (외부 이동 없이 미니앱에서 생성 완결)
- [ ] 토스 로그인 / 인앱결제(디지털 상품) / 토스 광고 연동

## 10. 토스 미니앱 전환 시 필수 체크리스트

> 웹앱은 자유롭지만, 미니앱 전환 시 토스 정책을 반드시 맞춰야 한다.

- **생성형 AI 고지·표시**: 최초 사용 시 AI 사용 사실 고지 + 결과물에 AI 생성 라벨/워터마크 (위반 시 최대 3,000만 원 과태료)
- **외부 링크 제한**: 폰트 생성·다운로드가 미니앱 내에서 완결되어야 함. 외부 사이트로 보내 다운로드하는 구조 불가
- **자사 앱 설치 유도 금지**: "앱 설치하면 더 많은 기능" 류 문구·배너 금지
- **로그인**: 토스 로그인만 사용 가능
- **결제**: 디지털 상품(폰트 파일)은 **인앱결제**만 사용 가능 (외부 PG 불가)
- **광고**: 앱인토스 전면형/보상형/배너 광고만 가능 (외부 광고 네트워크 불가)
- **어뷰징 방지**: 같은 워크스페이스에서 "테마만 다른" 유사 앱 중복 출시 불가 (폰트/이모티콘은 목적·핵심기능이 명확히 다르면 별도 앱 가능)
