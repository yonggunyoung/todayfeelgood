# 02. AI 설계 — 미활용 vs 활용의 경계선

핵심 질문에 대한 답: **"AI 미활용 / AI 활용 / 유료화를 어떻게 가르는 게 좋은가."**

## 1. 원칙: 기능을 "비AI 기본형 + AI 고급형"으로 이중화한다

AI를 쓸지 말지를 기능 단위로 가르는 게 아니라, **같은 기능 안에서 비AI 기본형(무료·무제한)과 AI 고급형(한도제·프리미엄)을 둘 다 만든다.** 이래야 ① 무료 사용자의 코어 경험이 완결되고 ② AI 변동비가 통제되고 ③ 프리미엄 차별화가 자연스럽다.

## 2. 기능 매트릭스

| 기능 | 비AI 기본형 (변동비 0) | AI 고급형 (회당 과금 대상) |
|---|---|---|
| 재고 입력 | 퀵추가(검색+즐겨찾기 2탭), 바코드(식약처 공공 DB) | 영수증/사진/주문캡처 파싱 |
| 소비기한 | 품목별 기본값 사전 자동 제안 | — (AI 불필요) |
| 임박 알림 | 규칙 기반 푸시 | — (AI 불필요) |
| 레시피 추천 | **재료 교집합 매칭 + 모드별 정렬** (아래 3절) | "있는 재료로 새 요리 생성", 변형 제안 |
| 모드 지표 | 영양 DB 기반 단백질/칼로리 표시 | 맞춤 코멘트, 주간 식단 플랜 생성 |
| 차감 | 레시피 연동 원탭 차감, 퍼지 수량 | 즉흥 요리 사진 차감, 월간 정합성 체크 |
| 잔반 | 타이머, 우선 추천 | "잔반 활용 요리" 생성 |
| 장보기 | 부족 재료 집계 + 파트너스 링크 | 패턴 학습 선제 알림(Phase 2, 통계로도 가능) |
| 유튜브 레시피(Phase 2) | 임베드 + 수동 저장 | 영상 → 구조화 레시피 추출 |

**채팅형 "AI 셰프"** 는 순수 AI 기능으로, 프리미엄의 간판 역할.

## 3. 모드별 추천은 대부분 AI가 필요 없다 (중요)

추천의 기본형은 **큐레이션된 레시피 DB에 대한 정렬·필터 문제**다:

```
후보 생성: 보유 재료 ∩ 레시피 재료 (커버리지 ≥ 80% 우선)
점수 = w1·재료커버리지 + w2·임박재료 사용 + w3·잔반 사용 + w4·모드 점수
  운동 모드   : 모드 점수 = 단백질 밀도 (영양 DB 컬럼 정렬)
  자린고비 모드: 모드 점수 = 임박 재료 가중 + 추가 구매 비용 역가중
  산모 모드   : 금기 재료 포함 레시피 하드 제외 후 영양 가중
```

레시피 시드는 **식약처 조리식품 레시피 공공 DB**(약 1,100개, 재료·분량·영양정보 포함)로 시작 → 비용 0원, 영양 데이터 내장. AI는 이 위에 "생성·변형·대화"를 얹을 때만 쓴다.

## 4. 산모 모드 안전 설계 (법적 리스크 통제)

의료 정보 성격이 있으므로 LLM 자유 생성에 맡기지 않는다:

1. **금기·주의 식재료 목록은 수동 큐레이션 규칙**으로 박는다 (예: 임신 중 — 날생선·생고기, 리스테리아 위험 연성치즈, 알코올, 고카페인, 일부 약초류 / 수유 중 별도 목록). 출처: 식약처·학회 공개 자료 기반, 출처 표기.
2. 추천 파이프라인에서 **하드 필터가 AI보다 먼저** 적용되고, AI 생성 결과도 출력 전에 같은 필터를 한 번 더 통과시킨다 (이중 검증).
3. 화면 고지: "본 정보는 의학적 조언이 아니며, 임신·수유 중 식단은 담당 의료진과 상의하세요."
4. "치료", "효능" 등 의료 광고로 해석될 표현 금지 (표시광고법 리스크).

## 5. AI 파이프라인 구성

모든 AI 호출은 **서버(Cloud Functions) 경유** — API 키 보호 + 사용자별 사용량 미터링 + 남용 방지. 클라이언트 직접 호출 금지.

| 용도 | 특성 | 모델 후보(단가 ↓순) | 비고 |
|---|---|---|---|
| 영수증/사진/캡처 파싱 | 대량·단순·구조화 출력 | Haiku 4.5 ($1/$5 per MTok) | JSON 스키마 강제(structured outputs)로 파싱 안정화 |
| 레시피 생성·변형, AI 셰프 | 품질 민감 | Sonnet 4.6 ($3/$15) 또는 Opus 4.8 ($5/$25) | 품질 평가 후 결정. 시스템 프롬프트는 프롬프트 캐싱 적용 |
| 주간 식단 플랜 | 복잡 추론·저빈도 | Opus 4.8 | 회수 제한 기능이라 단가 여유 있음 |
| 유튜브 추출(Phase 2) | 긴 입력(자막) | Sonnet/Opus + 캐싱 | 자막 없으면 STT 추가 비용 — 한도 차감 2회분 처리 |

공통 설계:
- **structured outputs**(`output_config.format`, JSON 스키마)로 파싱 출력을 스키마 보장 → 후처리 코드 단순화
- **프롬프트 캐싱**: 시스템 프롬프트(재료 사전 요약, 출력 규칙)를 캐시 블록으로 고정 → 입력 비용 ~90% 절감
- 호출 1건 = `ai_usage` 컬렉션에 기록 (사용자, 유형, 토큰, 비용 추정) → 무료 한도 집행 + 단가 모니터링

## 6. 회당 비용 추정 (설계용, 환율 1,400원/$ 가정)

| 호출 유형 | 토큰 가정 | 모델 | 추정 비용 |

| 영수증 1장 파싱 | 입력 ~2K(이미지 포함)/출력 ~0.5K | Haiku 4.5 | ~$0.005 ≈ **7원** |
| 생성형 추천 1회 | 입력 ~2K/출력 ~0.8K | Sonnet 4.6 | ~$0.018 ≈ **25원** |
