# 📈 상용화·트렌드 참견쟁이 — 평가 보고서

> 평가단(코드 수정 없음). 대상: "획 폰트공방" 현 제품 + Phase 2 설계문서.
> 기준일 2026-06-02. 1인 운영·무료티어·비용0 기본 전제 반영.

## 한 줄 결론
**지금 제품은 "잘 만든 무료 장난감"이지 "돈 받는 도구"가 아니다.** 매출 트리거(워터마크/라이선스/풀포맷)와 측정(애널리틱스·OG 공유)이 **코드에 0% 존재**한다. 설계 문서는 정답에 가깝게 잘 써졌으나 **선행 무코스트 수익화(P0)와 그로스 계측을 먼저 깔지 않으면, AI·스티커로 가도 깔때기 바닥이 새는 채로 간다.**

## 1순위 수익/트렌드 액션
**P0 "단건 다운로드" 결제 깔때기를 지금 엔진만으로 켠다 = 워터마크(폰트 name 테이블 + PNG 표식) → 상업 라이선스 증서(PDF) → 풀포맷(TTF/OTF/WOFF2 + @font-face) 묶음, 인앱결제 추상화(`PaymentProvider`) 뒤에.** AI·한글 손글씨는 매출 검증 후. 동시에 **OG 공유 이미지 라우트 + 익명 애널리틱스**를 깔아 깔때기를 측정 가능하게 만든다(이게 없으면 우선순위 판단 자체가 불가능).

## 1. "돈 낼 가치" 현실성 평가표

| 항목 | 현 상태(코드 확인) | 유료 전환 트리거? | 평가 |
|------|------------------|------------------|------|
| 변형 폰트 WOFF/TTF 무료 다운로드 | **구현됨, 전량 무료, 워터마크 없음** | 없음 | 미끼로는 OK, 매출 0. 기존 분석("변형 무료론 약함")과 **정합** |
| 워터마크 제거 | **코드에 흔적 없음**(engine name테이블·PNG 둘 다 표식 없음) | (가장 마찰 적은 트리거인데 부재) | **P0 갭** |
| 상업 라이선스 증서 | 없음 | 사업자·디자이너 결제 핵심 동기 | **P0 갭** |
| 풀포맷(OTF/WOFF2/웹킷) | WOFF/TTF만. OTF/WOFF2/CSS킷 없음 | 실무 필수 포맷 | **P0 갭, 엔진 보유라 노력 低** |
| AI 손글씨 완성/한글 | 미구현(설계만) | 진짜 지갑 여는 동기 | P2, 원가·기술 리스크로 후순위 적절 |
| 결제/로그인 추상화 | **코드 없음** | (전 매출의 전제) | **P0 기반 갭** |

**준비도 요약**: P0 3종(워터마크 제거·라이선스·풀포맷)은 설계상 "노력 低·엔진 보유"인데 **실제 코드에는 단 한 줄도 없다.** 가장 빨리 매출 낼 수 있는 구간이 가장 비어 있음 = 최대 기회이자 최대 갭.

**정합성 판정**: 기획서/수익화 문서의 "변형 무료 다운로드만으로는 약하다"는 분석은 **맞다.** 다만 문서는 "P0를 먼저"라고 명시해 놓고 빌드 웨이브(A~F)에서 P0를 **Wave F(맨 끝)**에 둔 모순이 있다. 무코스트 매출이 가능한 P0를 AI·스티커보다 뒤로 미룬 건 **순서 오류.** 적어도 "워터마크+라이선스+풀포맷"은 스티커앱 신규 개발보다 앞서야 한다.

## 2. 차별화 / 트렌드

**강점(이미 구현, 트렌드 적중)**
- **괴상함(weirdness)·구불구불(waviness)·변주 갤러리 9종·무드 프리셋 9종**: "한 손글씨 폰트"가 아니라 "내가 굴려 만드는 표정"이라는 놀이성 = **숏폼/밈/Y2K·키치 타이포 트렌드**와 맞물림. 경쟁(Calligraphr·Fontself는 사용자가 다 그려야 함, Google Fonts는 변형 불가)과 명확히 다름.
- **이미지 효과(질감/무늬/색) + PNG 내보내기**: 이미 구현됨. **굿즈/스토리/썸네일 즉시 사용**이라는 트렌드 접점인데 — 지금은 깊숙이 숨어 있다(아래 3번).
- **너구리 마스코트 "획이"**: 브랜드 친근감·바이럴 캐릭터 자산. 스티커앱 마스코트로 재활용 여지.

**약점/위험**
- **차별점이 랜딩·OG·SEO로 안 새어나간다.** 괴상함/변주 같은 "남들과 다른 점"이 정작 검색·공유 표면에 노출이 약함. 좋은 기능을 안에만 숨긴 형국.
- 한글이 "공개폰트 변형(내 글씨 아님)"으로 정직하게 라벨되어 있어 신뢰는 좋으나, **한글 시장의 진짜 수요(내 글씨 한글 폰트)는 아직 0% 충족** — 이게 차별화의 본진인데 미구현.
- 스티커(Wave E)는 트렌드 정중앙(이모티콘/굿즈)이고 비용0·브라우저 합성으로 ROI 높음. 단 **신규 앱 = 1인 운영 부담**. 폰트 P0 매출 검증 전 신규 앱 착수는 리소스 분산 위험.

## 3. 전환 퍼널 / 그로스 분석

**의도된 퍼널**: 방문(SEO/공유) → 생성(프리뷰) → 다운로드 시도 → 게이트 → 결제 → 재방문.

**실제 코드 상태로 본 퍼널 누수**

| 단계 | 코드 현실 | 누수 |
|------|----------|------|
| 방문(유입) | sitemap/robots는 있음. **OG 동적 이미지 라우트 없음**(`opengraph-image` 부재) | 공유해도 썸네일 미끼가 안 뜸 → 바이럴 루프 끊김 |
| 생성/활성 | 실시간 프리뷰·디바운스·변주 우수 | (양호) |
| 공유 | PNG 저장 버튼은 FontPreview 깊숙이 존재, 파일명 `hwoek-specimen` | 워터마크/서비스명 표식 없음 → **공짜 광고 기회 상실**. 공유 페이지(`/s/[id]`) 없음 |
| 다운로드 | 즉시 무료 다운로드, 게이트 0 | 결제 트리거 0 |
| 결제 | 결제 코드 0 | — |
| 측정 | **애널리틱스 0(gtag/plausible/posthog 전무)** | 어느 단계가 새는지 **측정 불가** → 그로스 판단 불가 |

**가장 큰 빠진 후크 3개**
1. **계측 부재** — 깔때기 KPI(생성수·다운로드율·게이트도달률)를 못 본다. 1인 운영일수록 "데이터로 우선순위"가 생존선. **가장 먼저 깔 것**(쿠키리스 익명, 비용0 Plausible/Umami self-host 또는 단순 이벤트 카운터).
2. **OG 동적 이미지** — 생성 결과(괴상한 글자체 견본)를 OG 카드로 → "이거 어떻게 만들었어?" 유입 루프. SSR 기반 이미 있어 라우트만 추가하면 됨. **노력 中·유입 효과 高.**
3. **워터마크(서비스명) on 무료 산출물** — PNG/폰트에 옅은 "획 hwoek.app" 표식 = 바이럴 광고 + 제거 업셀 트리거. **하나로 두 마리.**

## 4. 랜딩 가치제안 진단

현재 랜딩(`app/page.tsx`)은 **"무엇을 하는가"는 잘 팔지만 "왜 이걸 써야 하나(차별·결과·용도)"는 약하다.**

| 요소 | 현재 | 개선 제안 |
|------|------|----------|
| 헤드라인 | "내 손으로 만드는 글씨체" | 정직하나 평범. **결과·용도 박힌 카피**: "남들 다 쓰는 폰트 말고, 내가 굴려 만든 글자" / "괴상하게도, 우아하게도 — 슬라이더로 빚는 글씨체" |
| 차별점 노출 | 괴상함·변주·이미지효과가 본문에 안 보임 | 히어로에 **"weirdness·물결·9가지 변주"** 시각 데모(움직이는 GIF/스페시먼) 직접 노출 |
| 용도(왜) | 없음 | "썸네일·스토리·굿즈·로고 텍스트에" 같은 **사용 맥락** 1줄 |
| CTA | "만들러 가기"·"어떻게 만드나요?" | 양호. 다만 결과 공유 CTA("내 글자 PNG로 자랑하기") 추가로 바이럴 유도 |
| 신뢰 | 정직성 고지 우수 | 유지(차별 자산). 단 "무료" 강조 + 추후 "상업용은 라이선스 제공" 한 줄로 결제 인지 심기 |

핵심: **랜딩이 "놀이"만 팔고 "결과물의 쓸모"를 안 판다.** 트렌드 사용자(숏폼·굿즈 제작자)에게 "이걸로 뭘 만들 수 있나"를 보여줘야 전환 모수가 커진다.

## 5. 토스 미니앱 정합 체크

| 토스 제약 | 현 코드/설계 | 판정 |
|-----------|------------|------|
| 인앱결제만 | 결제 코드 0. 설계는 `PaymentProvider` 추상화 권장 | **추상화부터 깔면 정합.** 결제 짤 때 첫 줄부터 인터페이스로 |
| 외부 광고망 금지 | 광고 0, 기능 과금 지향 | 정합 |
| 앱 내 완결(외부 다운로드 금지) | 웹은 직다운(앱 내 Blob 저장). 미니앱은 보관함/공유 경로 분기 필요 | 설계 인지함. 다운로드도 provider 추상화 권장 |
| AI 생성 라벨 의무 | AI 미구현. `generatedBy` 필드 존재("baseFontVariation") | **선제 정합 우수.** AI 추가 시 `"ai"` 유니온 확장만 |
| 로그인=토스 | 로그인 0. `AuthProvider` 추상화 권장 | 결제 도입 시 같이 추상화 |
| 유사앱 중복 | 폰트=파일생성, 스티커=이미지세트로 핵심기능 구분 | 스티커 별도 앱 권고는 규정상 타당 |

**정합성 양호.** 단 "추상화는 코드 짤 때부터"가 핵심인데 아직 결제·로그인 코드가 없으니, **P0 결제를 시작하는 순간이 추상화를 강제할 마지막 적기.** 여기서 토스용 인터페이스로 안 짜면 나중 재작업.

## 6. 우선순위 — 가치·노력·수익 매트릭스 (상용화 관점 재정렬)

> 설계문서의 Wave 순서를 **상용화/측정 관점으로 재배치.** 핵심 수정: P0 매출과 계측을 스티커 신규앱보다 앞으로.

| 순위 | 액션 | 가치 | 노력 | 수익기여 | 트렌드 | 비고 |
|------|------|------|------|---------|--------|------|
| **C0** | 익명 애널리틱스 + 핵심 이벤트(생성/다운로드/게이트) | 高(판단근거) | 低 | 간접(전부) | — | **가장 먼저.** 비용0 self-host. 없으면 우선순위 추측 게임 |
| **C0** | OG 동적 이미지 라우트 + 무료 산출물 워터마크(서비스명) | 高(유입) | 中 | 간접+업셀 | 高(공유) | 바이럴 루프 + 제거 업셀 동시 |
| **C1** | P0 결제: 워터마크 제거·상업 라이선스 PDF·풀포맷(OTF/WOFF2/CSS킷) | 高 | **低**(엔진 보유) | **中~高** | — | **무코스트 첫 매출.** 단건 3,900~5,900원 |
| **C1** | `PaymentProvider`/`AuthProvider`/다운로드 provider 추상화 | 기반 | 中 | (전제) | — | 토스 동시대비, 재작업 방지 |
| **C2** | 랜딩 가치제안·용도·차별 재카피 + 결과공유 CTA | 中 | 低 | 간접 | 中 | 전환 모수 확대 |
| **C3** | Wave D 부분 스타일 추출(quick win) → 글리프 캡처(내 글씨) | 高 | 中~高 | 高 | 中 | "진짜 내 글씨" 차별 본진 |
| **C4** | Wave E 스티커앱(비AI, 브라우저 합성) | 매우高 | 中 | 中~高 | **매우高** | 이모티콘/굿즈 트렌드 정중앙. **단 P0 매출 검증 후 착수**(1인 분산 방지) |
| **C5** | AI 손글씨 완성(라틴→한글) 크레딧 과금 | 매우高 | 매우高 | 매우高 | 中 | 원가전가 검증 후. 메모리·라이선스 리스크 |

**실행 순서 요약**: 계측·OG·워터마크(C0) → P0 결제+추상화(C1) → 랜딩(C2) → 내 글씨(C3) → 스티커(C4) → AI(C5).
**문서와의 차이**: 설계문서는 디자인→스타일→한글→내글씨→스티커→수익화(F) 순. 본 평가는 **수익화 P0와 계측을 한글/스티커보다 앞으로** 당긴다. 이유: 무코스트로 매출 가능 + 측정 없이 다음 투자 판단 불가 + 1인 운영은 "검증된 매출" 후에만 신규앱 확장이 안전.

## 7. 1인 운영 현실 반영 메모
- **신규 앱(스티커)은 매력적이나 운영 표면을 2배로 늘린다.** 폰트 P0가 단돈이라도 매출을 내는지 먼저 본 뒤 확장. 검증 전 두 앱 동시 운영은 1인에게 과부하.
- **AI는 마지막.** 외부 API 후불·메모리·라이선스·"AI 생성" 라벨 모두 리스크. 크레딧 선결제 구조 전제 필수(설계문서 정합).
- **계측이 곧 레버리지.** 1인일수록 "어디가 새는지"를 데이터로 봐야 한정된 시간을 옳게 쓴다. C0를 미루면 나머지 우선순위가 전부 추측이 된다.
