# 한글 폰트 생성 전략 (Phase 2 아이디어)

> 작성: 한글 전략 아이디어 에이전트 / 코드 수정 없음. 이 문서는 빌더 착수용 설계 메모다.
> 현재 코드 기준: 엔진은 Recursive 가변폰트를 `wght/slnt/CASL`로 인스턴스화 → 라틴 a–z 서브셋 → WOFF/TTF.
> 계약은 `packages/core/src/index.ts`의 `FontParams = {weight, slant, curvature}` 단일 출처.

---

## 한 줄 결론 + 1순위 권장 MVP

**마법은 없다. 한글 11,172음절을 "내 글씨"로 직접 만드는 건 비AI로 불가능하고, AI로도 비싸다.**
1순위 권장: **(c) 기존 OFL 한글 가변/오픈폰트(Pretendard·Noto Sans KR 등)를 베이스로 라틴과 똑같이 슬라이더(굵기/기울기/곡률) 변형** — "한글도 슬라이더로 바꿔서 내보내기"를 가장 빠르게(비용 0, 기존 엔진 거의 재사용) 붙인다. 이건 "내가 그린 글씨"가 **아니므로** UI에 "기성 폰트 기반 변형"이라고 정직하게 고지한다. 그다음 2순위로 **(a) 자모 조합**을 붙여 "사용자가 그린 자모로 만드는 진짜 내 한글"을 연다.

## 1. 접근법 비교

한글 음절은 유니코드 `AC00`(가)~`D7A3`(힣), 총 11,172자 = 초성 19 × 중성 21 × (종성 27 + 받침없음 1)이다. 라틴 26자 세계관과 규모 자체가 다르다.

| 접근법 | 사용자 입력 노동 | 난이도 | 운영비용 | 결과 품질 | "진짜 내 글씨" 여부 |
|---|---|---|---|---|---|
| (a) 자모 조합 | 자모 낱자 ~67자(아래 산정) | 높음(조판 규칙·변형) | 0 (로컬) | 중. "조합 티"가 남음 | 부분적 (획은 내 것, 배치는 합성) |
| (b) 완성형 직접 입력 | 2,350~11,172자 | 낮음(기술은 단순) | 0 | 최상(전부 내 글씨) | 완전 |
| (c) 기성 OFL 폰트 변형 | 0자 | **낮음(엔진 재사용)** | 0 (로컬) | 상(원본 폰트 품질) | **아님** (기성 폰트) |
| (d) AI few-shot 스타일전이 | 자모/소수 음절 ~20–100자 | 매우 높음 | 유료(GPU/API) | 상(잠재적 최상) | 거의 (생성·근사) |

### (a) 자모 조합 (component assembly)
사용자가 **자모 낱자 세트**만 그리면, 초/중/종 위치 규칙으로 11,172음절을 합성한다. 비AI MVP의 "진짜 내 글씨" 핵심.

- **최소 입력 글자 수(가정 포함)**:
  - 단순 모델: 초성 19 + 중성 21 + 종성 27 = **67자**. (자모를 한 벌만 받는 안)
  - 현실 모델: 초성·종성은 **중성 모양에 따라 자형/위치가 달라진다.** 한글 조판은 보통 초성을 세로모임(ㅏ,ㅓ…)/가로모임(ㅗ,ㅜ…)/섞임모임(ㅘ,ㅢ…)으로 8벌(또는 그 이상)로 나눈다. 받침 유무에 따라서도 폭·높이가 바뀐다.
  - 따라서 "정말 자연스럽게" 하려면 초성 19 × (받침유무 2 × 모임유형 ~3) ≈ 수백 벌 입력이 필요 → 비현실적.
  - **타협안(권장)**: 입력은 67자 한 벌만 받고, **변형(스케일·이동·약한 가로/세로 압축)은 엔진이 규칙으로 처리**한다. "옛한글 수준의 완벽한 8벌 조판"은 포기하고 "읽을 만한 합성"을 목표로 한다. (가정: 품질은 (b)보다 낮고, 첫인상에서 "조합 폰트 티"가 난다.)
- **받침 유무·중성 형태별 초성 변형 문제(핵심 난점)**:
  - `가`(받침X, 세로모임)와 `각`(받침O)에서 초성 ㄱ의 높이·위치가 다르다.
  - `고`(가로모임)에서는 초성 ㄱ이 위로 가고 중성 ㅗ가 아래에 깔린다 → 초성 폭/위치가 또 다르다.
  - 비AI로는 이걸 **이미지 워핑이 아니라 좌표 변환(affine: scale + translate)** 으로 근사한다. 완벽하진 않지만 "읽힘" 수준은 가능.
- **모아쓰기 좌표/스케일 규칙(엔진 구현 골격, 가정)**:
  - EM 1000 기준 가상 격자. 음절을 모임유형으로 분류:
    - 세로모임(ㅏㅑㅓㅕㅣ…): 초성 왼쪽(폭 ~55%), 중성 오른쪽(폭 ~45%), 받침은 하단 전폭.
    - 가로모임(ㅗㅛㅜㅠㅡ): 초성 상단(전폭), 중성 하단, 받침 최하단.
    - 섞임모임(ㅘㅙㅚ…): 초성 좌상, 중성 우+하, 받침 최하단.
  - 각 칸에 자모 글리프를 affine으로 맞춰 넣고, `glyf` 컴포넌트(합성 글리프) 또는 펜으로 외곽선 합성.

### (b) 완성형 직접 입력 — 비현실적 노동량
- KS X 1001 상용 2,350자만 받아도, 사용자가 한 자에 5초씩 그려도 약 **3.3시간 연속 노동**. 11,172자 전체는 비현실(15시간+).
- 기술 난이도는 오히려 낮다(라틴 파이프라인에 글자 수만 늘림). 문제는 **사람**. → 단독 MVP 부적합. 단, (a)/(d)의 **보정 입력**(특정 음절만 사용자가 다시 그려 덮어쓰기)으로는 유용.

### (c) 기성 OFL 폰트 변형 — 가장 빠른 비AI MVP
- 라틴에서 Recursive VF에 한 것을, 한글 가변폰트에 그대로 적용.
- **후보 폰트(라이선스 확인 필요, 가정)**:
  - **Pretendard** (OFL, 가변, wght 축 보유) — 한/영 통합, 현대적. 1순위 후보.
  - **Noto Sans KR / Noto Serif KR** (OFL, 가변 wght) — 글자 커버리지 최고.
  - 나눔 계열(나눔고딕 등)은 라이선스가 OFL이 아닌 자체 라이선스인 경우가 있어 **재배포·임베딩 조건 개별 확인 필요**(가정).
- **곡률(curvature)**: 한글 가변폰트에는 Recursive의 CASL 같은 둥글기 축이 거의 없다. → 한글에서 curvature는 (i) 사용 안 함, (ii) wght와 함께 약하게 매핑, (iii) 후처리 라운딩으로 근사. **MVP는 "한글은 curvature 비활성/무시"가 정직하고 안전.**
- 출력은 기존과 동일: 음절 서브셋 → WOFF/TTF.
- **한계 고지 필수**: 결과물은 사용자 글씨가 아니라 "기성 폰트를 굵기/기울기로 변형한 것". 라이선스상 재배포 가능해도 "내 폰트"라 부르면 안 됨.

### (d) AI few-shot 스타일 전이 — Phase 2 유료, 기본 OFF
- 자모/소수 음절만 받아 나머지를 생성. "내 글씨 한글"의 진짜 해답이지만 비싸다.
- **모델 후보(가정, 2026 기준 인지)**:
  - 한글 특화 오픈 연구: **zi2zi / DG-Font / FontDiffuser** 계열(few-shot 폰트 생성). 한자/한글 글리프 생성에 자주 인용.
  - 확산모델 기반 글리프 생성(글자 이미지 → 벡터화) 또는 윤곽선 직접 생성 모델.
- **비용/리소스**: GPU 필수(무료티어 CPU로는 추론도 부담). 외부 API 또는 별도 GPU 인스턴스 → CLAUDE.md 비용철칙대로 **환경변수 OFF 기본 + 유료 사용자 전용 + "AI 생성" 라벨 강제**.
- **이미지→벡터** 단계(Potrace/OpenCV)에서 품질·노이즈 이슈. Phase 2 후반 연구 항목으로만.

## 2. 권장 단계적 MVP

### 단계 H0 — (c) 기성 폰트 한글 슬라이더 변형 [1순위, 즉시 착수]
- **정의**: OFL 한글 가변폰트를 베이스로, 기존 `weight/slant`(+ 한글에선 curvature 무시) 슬라이더로 변형 → 한글 음절 서브셋 WOFF/TTF 출력.
- **작업 범위(빌더)**:
  - Engine: `font_loader`에 한글 베이스 폰트(Pretendard VF 등) 추가 다운로드/캐시. `generator`에 `script` 분기 추가(라틴 베이스 vs 한글 베이스 선택, 또는 한/영 통합폰트 1개).
  - 서브셋 텍스트를 라틴 `TARGET_CHARSET`에서 **KS X 1001 상용 2,350자 우선 세트**로 확장(아래 3절).
  - 곡률 축이 없으면 한글 요청에서 curvature는 받되 무시(라틴 imagePng 무시와 동일 패턴).
  - Frontend: 미리보기 텍스트에 한글 샘플 추가, "기성 폰트 기반 변형" 고지 배지.
- **완료 기준**:
  - `POST /generate`에 `script: "hangul"`(또는 charset 옵션) → 한글이 포함된 WOFF가 나오고 프리뷰에서 한글이 슬라이더로 굵어/기울어진다.
  - 서브셋 2,350자 WOFF 크기·생성시간이 무료티어에서 허용 범위(아래 메모리 절 기준).
  - UI에 "이 결과는 내가 그린 글씨가 아니라 공개 폰트(○○ OFL)를 변형한 것"이라는 고지 노출.

### 단계 H1 — (a) 자모 조합으로 "내 글씨 한글" [2순위]
- **정의**: 사용자가 자모 낱자 67자(초19·중21·종27)를 캔버스로 그림 → 엔진이 모아쓰기 규칙으로 11,172음절(또는 상용 2,350자) 합성 → WOFF/TTF.
- **작업 범위(빌더)**:
  - Frontend: 자모 입력 캔버스(낱자 그리기 그리드 67칸), 진행률 표시, 부분 입력 허용.
  - Engine: 자모 벡터화(Potrace) → 모임유형 분류 → affine 배치 → 합성 글리프 빌드(fontTools `glyf` 컴포넌트 또는 펜).
  - "보정 입력": 자주 쓰는 음절 일부를 사용자가 다시 그려 덮어쓰기(선택).
- **완료 기준**:
  - 67자만 그려도 "내 글씨" 한글 음절이 읽히는 수준으로 합성되어 WOFF로 나온다.
  - 받침 유/무, 가로/세로모임에서 초성이 격자 안에 자연스럽게 배치(겹침·삐져나옴 없음).
  - UI에 "초성 ○자, 중성 ○자만 직접 그렸고 나머지 음절은 자동 조합"이라고 합성 비율 고지.

### 단계 H2 — (d) AI few-shot [후순위, 유료·기본 OFF]
- 자모 입력에서 더 자연스러운 합성을 AI로. 환경변수 OFF 기본, 유료 사용자만, "AI 생성" 라벨. Phase 2 후반 연구.

## 3. 기술 스펙 (fontTools 한글)

### 유니코드 음절 ↔ 자모 분해/합성
- 음절 코드 `S`(0xAC00 시작): `S - 0xAC00 = (초 × 21 + 중) × 28 + 종`.
  - 초성 index `cho = (S-0xAC00) // 588`, 중성 `jung = ((S-0xAC00) % 588) // 28`, 종성 `jong = (S-0xAC00) % 28` (jong=0이면 받침 없음).
- 자모조합(a)에서는 이 역산으로 어떤 낱자 글리프를 어느 칸에 넣을지 결정.
- 호환 자모(U+3131~) ↔ 조합용 자모(초성 U+1100~, 중성 U+1161~, 종성 U+11A8~) 구분 주의: 합성·입력 매핑에 조합용 자모 인덱스를 쓰는 게 표준.

### cmap / 서브셋 전략
- **상용 우선 2,350자**: KS X 1001 완성형 음절 세트를 1차 서브셋 텍스트로. (전체 11,172자 대비 ~21%로 크기·메모리 절감.)
- 단계적: 2,350자 → 필요 시 KS X 1001 외 자주 쓰는 음절 추가 → 풀세트는 다운로드 전용 옵션.
- `Subsetter.populate(text=...)`에 음절 문자열을 직접 넘기면 cmap·glyf가 알아서 정리됨(라틴과 동일 메커니즘). 자모조합(a)에서는 우리가 만든 합성 글리프를 cmap에 음절 코드로 연결.
- 한/영 통합폰트(Pretendard)면 라틴+한글을 한 파일에서 같이 서브셋 가능 → 앱 통합 단순.

### 라이선스 (OFL 재배포 조건)
- **OFL 핵심**: (1) 폰트/파생본 재배포 시 OFL 사본 동봉, (2) **Reserved Font Name(RFN)** 이 지정돼 있으면 그 이름을 파생 폰트 이름에 사용 금지(우리는 `UserFont-xxxx`처럼 자체 패밀리명으로 빌드하므로 충돌 회피), (3) 단독 판매 금지(앱·번들 일부로는 가능), (4) 라이선스/저작권 고지 유지.
- 실무: 생성 WOFF의 `name` 테이블에 출처/라이선스 고지를 넣거나, 다운로드 패키지에 OFL.txt 동봉. **Pretendard/Noto는 OFL 확인됨(가정, 배포 직전 원본 라이선스 파일 재확인 필수).** 나눔 계열은 OFL 아닐 수 있어 개별 확인.

### 메모리 (무료티어 부담)
- 한글 가변폰트 원본은 라틴(Recursive ~3.6MB)보다 큼: Pretendard VF, Noto Sans KR VF는 수~십수 MB대(가정). startup 1회 로드 + 디스크 캐시는 기존 `font_loader` 패턴 그대로 OK.
- 위험은 **요청당 `instantiateVariableFont` + 서브셋**의 일시 메모리. 음절 2,350자 서브셋은 라틴보다 글리프 수가 많아 피크가 커진다. → 동시성 세마포어(현재 3)를 **한글 요청은 더 낮게(예: 1~2)** 두거나, 한글 생성 결과를 파라미터 해시로 디스크/메모리 캐시.
- `MAX_DOWNLOAD_BYTES`(현재 10MB)를 한글 폰트 크기에 맞게 상향 검토.

## 4. 파라미터 계약 영향 (`packages/core/src/index.ts`)

> 하위호환 유지: 기존 라틴 클라이언트가 `script` 없이 보내면 `"latin"`으로 동작하게 한다. TS 변경 시 Python(`generator.py`/`main.py`)도 함께 갱신(계약은 단일 출처지만 Python은 직접 import하지 않음).

```ts
/** 타깃 스크립트. 미지정 시 기존 동작(라틴) 유지. */
export type Script = "latin" | "hangul";

/** 한글 생성 방식. H0=기성폰트 변형, H1=자모조합. */
export type HangulMode = "baseFontVariation" | "jamoAssembly";

export interface FontParams {
  weight: number;
  slant: number;
  curvature: number;        // 한글 baseFontVariation에서는 무시될 수 있음(고지)
  script?: Script;          // 기본 "latin"
}

/** (a) 자모조합 입력 모델 — H1 단계에서 사용 */
export interface JamoInput {
  /** 조합용 자모 인덱스 → 사용자가 그린 PNG(dataURL). 누락분은 베이스로 폴백. */

  imagePng?: string | null;          // 기존(라틴 스타일 참고용 자리)
```

## 5. 리스크 / 오해 방지 (정직성)

- **"내 글씨 한글"이라는 오해가 가장 큰 리스크.** (c)는 기성 폰트 변형이지 사용자 글씨가 아니다. UI에 반드시:
  - (c): "공개 폰트(○○, OFL)를 굵기/기울기로 변형한 결과입니다. 직접 그린 글씨가 아닙니다."
  - (a): "초성 N자·중성 N자만 직접 그렸고, 11,172음절은 규칙으로 자동 조합했습니다. 일부 글자는 어색할 수 있습니다."
- **곡률 무력화 고지**: 한글에서 curvature 슬라이더가 효과 없거나 약하면, 슬라이더를 숨기거나 "한글 미지원" 비활성 표시.
- **품질 기대치 관리**: (a) 자모조합은 "조합 폰트 티"가 본질적으로 남는다. 데모/마케팅에서 "전문가 손글씨 폰트급"으로 과장 금지.
- **라이선스**: 배포 직전 베이스 폰트 OFL 원본 재확인 + RFN 회피(자체 패밀리명) + 고지 동봉. 나눔 계열은 OFL 아닐 수 있음 → 사용 전 개별 확인.
- **AI 라벨(토스 대비)**: (d) 도입 시 결과물에 "AI 생성" 라벨/`generatedBy:"ai"` 강제, 환경변수 OFF 기본, 유료 전용.
- **메모리/DoS**: 한글 서브셋은 라틴보다 무겁다. 동시성 하향 + 결과 캐시 없이 무료티어에 풀면 OOM 위험.

## 요약 (빌더용)

- **1순위(H0)**: OFL 한글 가변폰트(Pretendard 등)를 베이스로 기존 슬라이더 변형 엔진 재사용 → "한글도 슬라이더로 변형" 즉시 출시. 비용 0, 난이도 낮음. 단 "기성 폰트 변형(내 글씨 아님)" 정직 고지 필수, 한글에서 curvature는 무시.

- **후순위(H2)**: AI few-shot(zi2zi/DG-Font/FontDiffuser 계열) — 유료·GPU·기본 OFF·"AI 생성" 라벨.

- (b) 완성형 직접 입력은 노동량(2,350자=3시간+) 때문에 단독 MVP 부적합, 보정 입력으로만.
